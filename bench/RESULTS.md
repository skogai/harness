# TOON vs JSON — measured token savings

Real token counts using `gpt-tokenizer` (OpenAI BPE cl100k_base). OpenAI's tokenizer is not identical to Claude's, but empirically tracks within a few percentage points — good enough to decide when TOON is worth it.

Generated by `node bench/run.mjs`. Regenerate workloads with `node bench/generate-workloads.mjs` (deterministic, seeded).

**Aggregate across all workloads:** 21,252 tokens saved (58.0%) over 36,627 JSON tokens.

**Benchmark gate:** pass — min aggregate savings 40.0%, encode failures 0.

## What this tells you

- **Tabular arrays of uniform objects** (users, transactions, logs, metrics) see 48–66% savings. This is the core use case.
- **Even small uniform arrays** (3 items, uniform schema) still win — roughly 56% on our sample. The TOON header cost is amortized fast.
- **Deeply nested irregular objects** see 40% savings on our sample — lower ceiling but still worthwhile if the payload is hot in prompts.
- Rule of thumb: if the data has any repeated object shape at all, TOON is probably worth trying. The only reliable loss case is very small non-uniform blobs where JSON is already terse.

## Methodology caveats

- Tokenizer: `gpt-tokenizer@3.x` (cl100k_base / OpenAI BPE). Claude uses a different tokenizer; exact Claude counts require a provider-specific token count endpoint.
- TOON: `@toon-format/toon@2.1.0`, default options (comma delimiter, key folding on).
- Workloads: deterministic, seeded PRNG. See `bench/generate-workloads.mjs` for the exact shape of each sample.
- These numbers are **input-token** savings only. Output tokens are unaffected.
